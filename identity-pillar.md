# Identity Pillar

> Zero Trust implementation guide for the Identity pillar: authentication, authorization, privileged access, and identity governance

**CISA Pillar:** Identity
**Core Question:** Is the right person authenticated with the right level of assurance for this specific resource at this specific time?

---

## Why Identity Is the Foundation

In a Zero Trust architecture, identity is the primary policy signal. The network can no longer be used as a proxy for trust. The question is not "is this request coming from inside the network?" but "who is this, what are they trying to access, and do they have a legitimate reason to access it right now?"

Identity is also the most abused attack vector. Credential-based attacks, including phishing, password spraying, MFA fatigue, and token theft, are the dominant initial access method in enterprise breaches. Every investment in identity hardening has a direct, measurable impact on your overall breach risk.

---

## Identity Maturity Stages (CISA Model)

| Control Area | Traditional | Initial | Advanced | Optimal |
|---|---|---|---|---|
| Authentication | Password only | MFA on privileged access | MFA enforced, phishing-resistant started | Phishing-resistant MFA everywhere, passwordless |
| Authorization | Static role-based access | Some attribute-based policies | Risk-based Conditional Access | Just-in-time, just-enough access |
| Privileged Access | Shared admin accounts | PAM deployed, some use | PAM enforced, session recording | Just-in-time PAM, full session monitoring |
| Identity Governance | Manual provisioning | Lifecycle management started | Automated provisioning and de-provisioning | Continuous entitlement review and certification |
| SSO | Minimal, siloed | SSO for major apps | SSO with all enterprise apps | Universal SSO including legacy apps |
| Service Accounts | Unmanaged, over-privileged | Inventory complete | Managed secrets, reduced scope | Workload identity, no long-lived credentials |

## Control 1: Multi-Factor Authentication

MFA is the single highest-impact identity control available. It blocks the overwhelming majority of credential-based attacks. It is non-negotiable in a ZT program.

### MFA Implementation Tiers

**Tier 1: SMS and voice OTP**
Provides significant protection over password-only authentication. Vulnerable to SIM swapping and real-time phishing (adversary-in-the-middle attacks where the attacker relays the OTP before it expires). Acceptable as a transition step but should not be the end state.

**Tier 2: Authenticator app (TOTP)**
Time-based one-time passwords generated by an app (Microsoft Authenticator, Google Authenticator). Stronger than SMS. Still vulnerable to real-time phishing and MFA fatigue attacks (push notification bombing). The current baseline for most enterprises.

**Tier 3: Push notification with number matching**
Push-based MFA where the user must enter a number displayed on the login screen into the authenticator app. Blocks MFA fatigue attacks where the attacker tries to trick the user into approving an unexpected push. Should be configured on all Microsoft Authenticator deployments.

**Tier 4: Phishing-resistant MFA (FIDO2/WebAuthn, Certificate-based)**
FIDO2 hardware security keys (YubiKey, Google Titan) or platform authenticators (Windows Hello for Business, Face ID). Cryptographically bound to the legitimate origin site. Cannot be relayed by a phishing site. This is the CISA-recommended end state for all high-value access.

### MFA Rollout Priority

| Priority | User Population | Rationale |
|---|---|---|
| 1 | Global Administrators and Privileged Roles | Highest blast radius if compromised |
| 2 | IT Administrators and Help Desk | Common pivot targets post-compromise |
| 3 | Finance, HR, Legal (sensitive data access) | High-value data, regulatory exposure |
| 4 | All employees | Foundation for ZT identity |
| 5 | Vendors and contractors | External access is inherently higher risk |
| 6 | Service accounts with interactive capability | Often overlooked, frequently targeted |

## Control 2: Conditional Access

Conditional Access is the policy engine that evaluates access requests in real time and enforces MFA, device compliance, location restrictions, and session controls based on the risk level of the request.

### Conditional Access Signal Sources

| Signal | Examples | Policy Response |
|---|---|---|
| User identity | Who is the user? What roles do they hold? | Require step-up MFA for privileged roles |
| Device state | Is the device managed? Compliant? Joined? | Block unmanaged devices from sensitive apps |
| Location | Named location, IP range, country | Require MFA from unknown locations; block from high-risk countries |
| Application | Which app is being accessed? Sensitivity level? | Require phishing-resistant MFA for admin portals |
| Sign-in risk | Real-time risk score from identity protection | Require MFA or block high-risk sign-ins |
| User risk | Aggregated risk from past behavior and detections | Force password reset or block high-risk users |
| Session context | Session duration, activity patterns | Enforce session limits and re-authentication |

### Conditional Access Policy Framework

**Policy 1: Require MFA for all users accessing cloud applications**
Scope: All users. Apps: All cloud apps. Grant: Require MFA. This is the baseline. No exceptions without documented risk acceptance.

**Policy 2: Require compliant or hybrid-joined device for sensitive applications**
Scope: All users. Apps: Finance, HR, CRM, admin portals. Grant: Require device compliance OR hybrid Azure AD join. This enforces device trust as a second signal alongside identity.

**Policy 3: Block legacy authentication protocols**
Scope: All users. Apps: All. Conditions: Client apps = Exchange ActiveSync, Other clients. Grant: Block. Legacy protocols do not support MFA. They must be blocked.

**Policy 4: Require phishing-resistant MFA for admin roles**
Scope: All privileged roles (Global Admin, Security Admin, etc.). Apps: All. Grant: Require authentication strength = Phishing-resistant. This targets your highest-value accounts.

**Policy 5: Sign-in risk policy**
Scope: All users. Conditions: Sign-in risk = High. Grant: Require MFA. Automated response to detected anomalous sign-in signals.

**Policy 6: User risk policy**
Scope: All users. Conditions: User risk = High. Grant: Require password change. Automated response to detected credential compromise.

**Common Gaps:**
- Break-glass accounts excluded from all policies (required, but must be monitored)
- Service accounts excluded without alternatives
- No session control policies for cloud app access after authentication
- Missing coverage for non-browser client applications

## Control 3: Privileged Access Management

Privileged access is the most critical attack surface in any enterprise environment. Administrative credentials provide the keys to every system. Their management requires dedicated controls beyond standard user authentication.

### PAM Core Capabilities

| Capability | Description | Priority |
|---|---|---|
| Privileged account vaulting | Storing and rotating admin credentials in a secure vault | Critical |
| Session isolation | Routing privileged sessions through a secure jump server or broker | Critical |
| Session recording | Recording all privileged sessions for audit and forensics | Critical |
| Just-in-time access | Granting elevated access for a specific time window and purpose | High |
| Least privilege enforcement | Ensuring admin accounts have only the permissions required for the task | High |
| Password rotation | Automated rotation of privileged account passwords after each use | High |
| Multi-person authorization | Requiring a second approver for the most sensitive operations | High |
| Approval workflows | Formal request and approval process for privileged access grants | Medium |

### Privileged Access Tiers (Microsoft PAW Model)

**Tier 0 (Control Plane):** Domain controllers, Azure AD, PKI, backup infrastructure, identity systems. Access from dedicated Privileged Access Workstations (PAWs) only. No internet access from PAWs. Phishing-resistant MFA required.

**Tier 1 (Management Plane):** Servers, cloud infrastructure, network devices, security tools. Access from managed and compliant devices. Standard phishing-resistant or strong MFA required.

**Tier 2 (User Plane):** Workstations, user applications. Standard MFA required.

The critical rule: higher tiers must never authenticate from lower-tier devices. An admin must never log into a domain controller from a workstation used for browsing or email.

### Service Account Management

Service accounts are frequently the weakest link in enterprise identity security. They are:
- Often over-privileged (created with admin rights for convenience)
- Rarely rotated (passwords set to never expire)
- Poorly inventoried (no owner, no documented purpose)
- Targeted specifically by attackers (Kerberoasting, DCSync)

**Minimum service account controls:**

1. Inventory every service account: name, purpose, owner, systems it accesses, last password change
2. Remove privileges not required for the documented function
3. Set password rotation on a defined schedule (90 days or less)
4. Where possible, replace service accounts with managed identities (Azure Managed Identity, gMSA for on-premises)
5. Disable interactive logon for service accounts
6. Monitor service account usage for anomalous patterns

## Control 4: Single Sign-On

SSO reduces authentication friction for users while centralizing identity governance. Without SSO, users authenticate independently to each application, creating unmanaged identity silos that cannot be governed, monitored, or de-provisioned consistently.

### SSO Integration Priority

**Priority 1: Connect all SAML 2.0 and OIDC-capable SaaS applications**
Modern SaaS applications support federated authentication. There is no valid reason for these to maintain independent credential stores. Connect them to your IdP.

**Priority 2: Legacy applications using Kerberos or NTLM**
On-premises applications using Windows-integrated authentication can be fronted by Azure AD Application Proxy or similar reverse proxy solutions to bring them into the SSO umbrella without re-architecting the application.

**Priority 3: Custom or in-house applications**
Prioritize by sensitivity and user population size. High-sensitivity apps (finance, HR) first.

**Priority 4: Third-party vendor portals and partner applications**
Federated B2B access where your IdP is used as the source of truth for vendor authentication.

## Control 5: Identity Governance

Identity governance ensures that access rights are accurate, documented, and reviewed. Without it, access accumulates over time and former employees, role changes, and project completions leave behind permissions that create unnecessary risk.

### Lifecycle Management

| Event | Required Action |
|---|---|
| New employee | Provision access based on role and department, not individual requests |
| Role change | Add new access for new role AND remove access from previous role |
| Employee offboarding | Disable account within 24 hours; revoke all sessions; transfer owned resources |
| Contractor/vendor onboarding | Grant minimum required access with defined expiration |
| Contractor/vendor offboarding | Immediate access removal on contract end date |

### Access Review Cadence

| Access Type | Review Frequency | Reviewer |
|---|---|---|
| Privileged roles (Global Admin, Security Admin) | Monthly | CISO or delegate |
| Sensitive application access (finance, HR, legal) | Quarterly | Application owner |
| All other application access | Semi-annually | Direct manager |
| External/guest access | Quarterly | Sponsoring employee |
| Service account access | Semi-annually | System owner |

## Identity ZT Checklist

Use this checklist to assess your current Identity pillar maturity:

- [ ] MFA enforced for all users on all cloud applications
- [ ] Number matching or additional context enabled on push-based MFA
- [ ] Legacy authentication protocols blocked
- [ ] Phishing-resistant MFA (FIDO2 or Windows Hello for Business) deployed for privileged roles
- [ ] Conditional Access policies covering device compliance, location, and risk
- [ ] PAM solution deployed with session recording for Tier 0 and Tier 1 access
- [ ] Just-in-time privileged access enabled via PIM or equivalent
- [ ] Service account inventory complete with owners documented
- [ ] Service accounts moved to managed identities or gMSAs where possible
- [ ] SSO connected for all major SaaS and cloud applications
- [ ] Automated provisioning and de-provisioning implemented
- [ ] Access reviews running on documented cadence
- [ ] Break-glass accounts defined, secured, and monitored
- [ ] Identity Protection or equivalent risk-based access enabled
- [ ] Passwordless authentication in production for at least privileged users
